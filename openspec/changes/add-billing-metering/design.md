# 设计：计费（token 按量计费）（已实现，回填记录）

## Context

- 对话由 `app/api/chat/route.ts` 经 `streamText` 调用真实付费模型（MiniMax 直连；DeepSeek/OpenAI 等经「模型&gateway」模块解析）。每次调用都会产生供应商侧的真实 token 成本，应用需要向用户收费以覆盖成本并保持利润。
- 供应商真实成本不会内联在流式响应里返回；只有走 Vercel AI 网关的调用事后可用 `generationId` 查到真实 USD 成本，且查询有结算延迟（不能即时拿到）。这决定了计费不能是「等真实成本再扣」的单阶段模型。
- 金额是货币场景，浮点误差在长期累加下不可接受，需要一个整数记账单位；供应商定价常以美元或人民币两种币种混杂，需要统一折算口径。
- 仓库已有 Drizzle + Postgres 基建（`lib/db/index.ts`、`lib/db/schema.ts`、`pnpm db:generate|migrate`），`thread_chat` schema 下新增计费两张表即可复用现有迁移与连接管理，无需另起基建。
- 本变更是**回填记录**：`constants/pricing.ts`、`lib/db/billing-schema.ts`、`lib/billing/credits.ts`、`app/api/chat/route.ts` 的计费集成、`app/api/billing/*` 均已实现并在跑；本文档把已落地的设计决策显式记录下来，供后续维护与「登录注册」「模型&gateway」「收款&支付」等相邻模块对齐边界。

## Goals / Non-Goals

**Goals:**

- 金额记账用整数单位，杜绝浮点误差；换算口径集中一处，不在各调用点重复计算。
- 保证向用户收取的价格相对供应商成本的利润率 ≥30%，且对多币种成本一视同仁地折算。
- 计费体验即时（发起对话前知道能不能发、调用结束立刻看到扣费），不因为等待真实成本而阻塞或挂起会话。
- 供应商成本一旦产生就必然被计费，即使客户端中途断连；扣费与流水写入不出现「一半」状态。
- 真实成本可用时能修正估算误差，且修正过程幂等、可安全重跑。
- 每次调用的用量与费用对用户可见（消息级 + 账户级）。

**Non-Goals:**

- 强一致的「预扣费/挂起」模型——见 D5，明确弃选。
- 初始额度发放的触发时机（哪个注册环节调用 `ensureUserCredits`）——留给「登录注册」模块。
- 模型可用性判定、provider 解析、网关配置——留给「模型&gateway」模块，本模块只消费其暴露的 `providerMetadata.gateway.generationId`。
- 实际收款、支付渠道 webhook、充值 UI——留给「收款&支付」模块，本模块只提供 `addCreditsMicros` 供其调用。
- 多币种向用户展示切换（目前统一以人民币「元/微元」呈现）。

## Decisions

### D1：金额用「微元」整数记账，换算集中在 `constants/pricing.ts`

**选择**：所有金额字段（余额、成本、售价）在内存与数据库中一律是整数「微元」（1 元 = 1,000,000 微元），`bigint` 存储；任何币种换算、加价计算、格式化展示，全部经由 `constants/pricing.ts` 的函数完成，不在调用点手写乘除。

**理由**：货币场景下浮点数的舍入误差会在高频累加（每次调用都扣费）下逐渐失真，且不同环节各自实现换算容易出现口径分歧（比如一处用了四舍五入、另一处用了截断）。把唯一事实来源收敛到一个文件的一组纯函数里，新增模型定价或调整汇率只改一处，`lib/billing/credits.ts`、`app/api/chat/route.ts`、`lib/billing/usage-meta.ts` 都只是调用方。

**弃选**：用 `number`/浮点元为单位——短期实现更直觉，但长期在对账、加总、展示三处都会暴露精度问题；引入专门的 decimal 库——对本项目的量级（token 定价、微元计费）是过度设计，整数微元已经足够精确且零依赖。

### D2：加价公式 `售价 = ceil(成本 / (1 - 0.3))`，成本按模型原生币种记录

**选择**：`PROFIT_MARGIN = 0.3`，售价对成本按 `ceil(成本微元 / (1 - PROFIT_MARGIN))` 计算，保证利润率 `(售价-成本)/售价 ≥ 30%`（向上取整只会让利润率更高，不会更低）。模型成本按供应商官方计价页的**原生币种**填入 `MODEL_COST`（如 `minimax-m2` 是 CNY，`deepseek-chat`/`gpt-4o-mini` 是 USD），美元成本再统一经 `USD_TO_CNY`（默认 7.3，环境变量可覆盖）折算为微元。

**理由**：原生币种记录避免了「先把美元估算成人民币价格再填表」这种二次近似——真实误差只应该来自汇率这一个可控参数，而不是分散在每条定价里的经验估算。`USD_TO_CNY` 集中在一处也让"汇率留缓冲、宁高勿低"这条业务纪律可执行：调高这一个常量，所有美元计价模型的成本估算同步上浮，利润底线不会被单个模型的汇率假设拖垮。

**弃选**：把美元成本手工近似换算成人民币后直接填进 `MODEL_COST`——相当于把汇率假设硬编码进每一条定价，汇率变化时要逐条核对修改，且不同模型的换算时间点不一致会引入不必要的误差分散。

### D3：两阶段计费——onFinish 即时估算扣费 + 事后按 generationId 对账修正

**选择**：调用结束（`onFinish`）时用价目表**估算**成本与售价，立即扣费并写入 `usage_records`（`cost_source='estimate'`）；随后由定时任务 `reconcilePendingCosts` 按该次调用的 `generationId`（若经网关）向 Vercel AI 网关拉取**真实** USD 成本，重算售价，用差额（delta）修正用户余额，并把该行标记为 `cost_source='gateway'`。

**理由**：网关不会在流式响应里内联回传真实成本，真实成本的可用时间点是事后的、不确定的。如果等真实成本才扣费，要么阻塞响应（不可接受，用户等的是回答不是账单），要么把会话挂起到「未结算」状态直到对账跑完（复杂度高，且余额在这段时间里对用户不可预测——他们看不到自己还剩多少额度）。价目表估算牺牲的是「与真实成本的微小偏差」，换来的是余额准实时可见、用户体验不被计费延迟拖累；偏差由后续对账兜底收敛，双方各司其职。

**弃选**：单阶段、等真实成本落地才扣费——延迟不可预期（网关结算窗口不受我们控制），且要为「调用已完成但费用未知」的中间态单独建模（挂起/冻结余额），复杂度远超两阶段方案换来的精度收益。

### D4：扣费与写流水在同一数据库事务内

**选择**：`chargeUsage` 里，`UPDATE user_credits ... SET balance_micros = balance_micros - price` 与 `INSERT INTO usage_records ...` 包在同一个 `db.transaction`；`reconcilePendingCosts` 里修正 `usage_records` 与按 delta 调整 `user_credits` 同样包在一个事务。

**理由**：这两步必须同生共死——只扣钱不记账，账单对用户不可解释也无法对账；只记账不扣钱，余额和实际消费永久脱节。事务是最直接、最不容易在未来维护中被破坏的保证方式，不需要额外补偿逻辑或对账任务去弥合两步之间可能出现的裂缝。

**弃选**：先扣费后异步写流水（如消息队列）——为一个本质是本地强一致的操作引入了不必要的最终一致性与失败重试复杂度。

### D5：后付费 + 并发竞态的敞口控制，而非强一致预扣

**选择**：发送前 `hasPositiveBalance` 只做粗粒度拦截（余额 >0 才允许发起新对话），不锁定/冻结额度；`chargeUsage` 允许把余额扣至负数，以覆盖「拦截时有余额、但生成过程中把余额耗尽」这最后一条消息的真实成本；`maxOutputTokens`（8192）给每次调用的最大产出加上限，从而给单次调用的最大成本设了天花板；`after(consumeStream)` 保证断连时计费依然发生（见 D3 相邻但独立的问题：断连不应该变成免费）。

**理由**：强一致预扣（发起前锁定预估最大成本、结束后按实际用量释放差额）能彻底消灭超支敞口，但代价是：预扣金额只能按 `maxOutputTokens` 的最坏情况估算（导致大多数请求会临时冻结远超实际花费的额度，体验很差——用户会看到余额莫名其妙地大幅下降又回升）；且需要给"已预扣但请求还没结束"这种中间态建模，复杂度不成比例。本设计接受「余额可能被扣至轻微负数」这一有限敞口，用 `maxOutputTokens` 把单次超支的上限锁死，用统计意义上的"负余额用户会被下一次 `hasPositiveBalance` 挡在门外"来收敛整体风险——这是在体验和风控之间的务实取舍，不是设计疏漏。

### D6：对账幂等 + `CRON_SECRET` 保护 + 15 分钟节奏

**选择**：`reconcilePendingCosts` 只扫描 `cost_source='estimate'` 且 `generation_id` 非空的行；每行处理成功后立即翻成 `'gateway'`，未就绪（网关返回 null）的行保持 `'estimate'` 留给下次重试。触发端点 `app/api/billing/reconcile/route.ts` 要求请求携带匹配 `CRON_SECRET` 的鉴权头，**未配置该环境变量时直接拒绝所有请求**（不存在"裸奔"的默认放行）。`vercel.json` 配置 `*/15 * * * *` 触发。

**理由**：幂等是对账任务的生命线——它会被定时反复调用，任何非幂等逻辑都会导致重复修正、余额被多次扣减或退回。「只认 estimate、成功翻 gateway」这个状态机足够简单，不需要额外的分布式锁或去重表。`CRON_SECRET` 未配置即拒绝（而不是未配置即放行）是故意的保守默认——一个能触发资金变动的端点不应该在忘记配置密钥时对外暴露。15 分钟是网关结算延迟与"额度偏差不宜挂太久"之间的折中，不是精确推导值。

### D7：`usage_records.thread_id` 不加外键

**选择**：`usage_records.threadId` 是普通文本列，不引用 `threads.id`，也没有级联约束。

**理由**：账单流水是财务记录，其生命周期不应该被对话数据的生命周期左右——用户删除一个对话（thread）是很平常的操作，不应该连带删掉已经发生的计费历史，否则账单会对不上、对账会失去依据。反过来，账单表也不应该阻止用户删除对话（如果加了外键且不设级联，删除 thread 会因为存在关联的 usage_records 而失败）。两者故意解耦，代价是 `threadId` 可能引用一个已经不存在的对话，但这是可接受的——`threadId` 在这里只是一个「用于聚合展示」的标签，不是需要引用完整性的关系。

## Risks / Trade-offs

- **[估算与真实成本的短暂偏差]** → 在对账（≤15 分钟节奏）修正前，用户看到的余额基于估算而非真实成本；由 D2 的定价表口径（成本只应低估不应高估，汇率宁高勿低）与两阶段对账机制共同收敛，偏差窗口短且方向可控。
- **[并发多请求透支]** → 发送前拦截只在请求发起的那一刻检查一次；如果同一用户几乎同时发起多个请求，多个请求都可能通过 `hasPositiveBalance` 检查后才依次扣费，理论上可以透支到比单次 `maxOutputTokens` 上限更负的余额。v1 接受这一敞口（见 D5 的权衡），未来若要收紧可以在 `hasPositiveBalance` 处加悲观锁或改为条件更新，本变更不预支这一复杂度。
- **[直连模型永不产生 `generationId`，对账永远跳过]** → MiniMax 直连等不经网关的调用没有 `generationId`，`usage_records` 该字段为空，`reconcilePendingCosts` 的查询条件 (`isNotNull(generationId)`) 天然跳过这些行，它们永远停留在 `cost_source='estimate'`。这是预期行为而非缺陷：没有网关就没有可对账的真实成本来源，估算即最终结果。
- **[`CRON_SECRET` 未配置导致对账静默失效]** → 对账端点在未配置密钥时直接拒绝而非报错闹钟，运维如果忘记配置该环境变量，对账会一直不跑但不会有明显告警（只有主动检查 `usage_records` 里 `estimate` 行堆积才会发现）。记录在此，作为部署检查清单的一项。
