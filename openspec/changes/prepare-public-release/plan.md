# Public Release Implementation Plan

## 1. Interface and Type Contracts

This section is normative for implementation. Public modules and component
boundaries are defined before file-level work so presentation changes do not
reintroduce route strings, provider details, or duplicated content.

### 1.1 Project identity and external links

Add `constants/project.ts` as the single TypeScript source for product identity
used by the landing page:

```ts
export const PROJECT = {
  name: "Thread Chat",
  liveUrl: "https://thread-chat.zilin.im/",
  repositoryUrl: "https://github.com/hifizz/thread-chatbot",
  copyrightHolder: "hifizz",
  copyrightYear: 2026,
  licenseSpdx: "AGPL-3.0-only",
} as const
```

**Input:** none.

**Output:** immutable public product metadata. This object MUST NOT contain
secrets, environment-specific credentials, pricing, or deployment IDs.

Markdown documents duplicate these public literals intentionally because GitHub
does not execute TypeScript; application code MUST import the constants.

### 1.2 Application route contract

Extend `constants/routes.ts`:

```ts
export const ROUTES = {
  landing: "/",
  startChat: "/start-chat",
  flagship: "/thread-chat",
  signIn: "/sign-in",
  account: "/account",
} as const

export function threadTreeRoute(treeId: string): `/thread-chat/${string}`
```

`threadTreeRoute` MUST be the route builder used by the new start page.

**Input:** a UUID tree ID already generated by the caller.

**Output:** an application-relative thread-tree URL.

**Errors:** none; UUID validation remains the responsibility of the destination
page and server API. The builder performs formatting only.

### 1.3 Landing content types

Replace the placeholder-oriented contract in `constants/landing.ts` with content
types matching the approved product story:

```ts
export type LandingSectionId =
  | "how-it-works"
  | "workspace"
  | "canvas"
  | "capabilities"

export interface LandingNavItem {
  label: string
  href: `#${LandingSectionId}` | string
  external?: boolean
}

export interface LandingCta {
  label: string
  href: string
  external?: boolean
  accessibleLabel?: string
}

export interface LandingHeroContent {
  eyebrow: string
  title: string
  subtitle: string
  primaryCta: LandingCta
  secondaryCta: LandingCta
}

export interface LandingStep {
  number: "01" | "02" | "03"
  verb: "Select" | "Branch" | "Navigate"
  title: string
  description: string
}

export interface LandingShowcaseContent {
  id: Extract<LandingSectionId, "workspace" | "canvas">
  eyebrow: string
  title: string
  description: string
  notes: readonly string[]
}

export type LandingCapabilityIcon =
  | "messages"
  | "database"
  | "fileText"
  | "search"

export interface LandingCapability {
  icon: LandingCapabilityIcon
  title: string
  description: string
}

export interface LandingContent {
  nav: readonly LandingNavItem[]
  hero: LandingHeroContent
  steps: readonly LandingStep[]
  workspace: LandingShowcaseContent
  canvas: LandingShowcaseContent
  capabilities: readonly LandingCapability[]
  closing: {
    title: string
    description: string
    primaryCta: LandingCta
    secondaryCta: LandingCta
  }
}
```

`LANDING` MUST satisfy `LandingContent`, use `ROUTES.startChat` for both primary
calls to action, and use `PROJECT.repositoryUrl` for GitHub actions.

**Input:** route and project constants.

**Output:** immutable English landing content.

**Invariant:** application components render content but do not own product copy
or external URLs.

### 1.4 Landing component interfaces

All landing components remain server components. Shared props:

```ts
export interface LandingSectionProps {
  className?: string
}

export interface LandingCtaLinkProps {
  cta: LandingCta
  tone: "primary" | "secondary" | "text"
  className?: string
}
```

Planned component contracts:

```ts
export function LandingHeader(): ReactElement
export function Hero(props: LandingSectionProps): ReactElement
export function InteractionSteps(props: LandingSectionProps): ReactElement
export function WorkspaceShowcase(props: LandingSectionProps): ReactElement
export function CanvasShowcase(props: LandingSectionProps): ReactElement
export function CapabilityGrid(props: LandingSectionProps): ReactElement
export function ClosingCta(props: LandingSectionProps): ReactElement
export function LandingFooter(): ReactElement
export function LandingCtaLink(props: LandingCtaLinkProps): ReactElement
```

**Input:** immutable content from `LANDING` and optional layout class names.

**Output:** semantic React elements only.

**Side effects:** none. No component reads session state, local storage, window
state, or remote data.

`LandingCtaLink` MUST render internal destinations with `next/link`; external
destinations MUST open normally with safe external-link attributes when a new
tab is requested. Every icon-only decoration MUST be `aria-hidden`; calls to
action keep textual accessible names.

### 1.5 Landing visual module boundary

Create `components/landing/landing.module.css`.

**Input:** component class assignments and a root `.page` class.

**Output:** landing-scoped paper, ink, highlight, annotation, branch-line,
depth-color, responsive, focus-visible, and reduced-motion presentation.

**Invariant:** the module MUST NOT mutate `:root`, `.tc`, shared shadcn tokens, or
non-landing elements. Decorative textures MUST use CSS gradients/pseudo-elements
and MUST NOT require a runtime animation dependency.

### 1.6 Fresh-chat server entry

Add `app/start-chat/page.tsx`:

```ts
export const metadata: Metadata

export default async function StartChatPage(): Promise<never>
```

Behavioral pseudocode:

```ts
const session = await getSession()
if (!session) {
  redirect(signInWithRedirect(ROUTES.startChat))
}
redirect(threadTreeRoute(randomUUID()))
```

**Input:** the current authenticated request/session.

**Output:** an HTTP redirect only; the route has no visible page state.

**Unauthenticated output:** redirect to `/sign-in?redirect=%2Fstart-chat`.

**Authenticated output:** redirect to a newly generated
`/thread-chat/{validUuid}` on every request.

**State mutation:** none before the destination mounts. The existing thread-tree
loader creates/persists the tree using its current lifecycle.

**Cache contract:** the route MUST execute per request and MUST NOT reuse a
previous UUID redirect.

## 3. Implementation Sequence

1. Read the relevant bundled Next.js 16 documentation for server redirects,
   request dynamism, metadata, CSS modules, and links before editing App Router
   files.
2. Add project identity, route, and landing content contracts. Run
   `pnpm typecheck`.
3. Add `/start-chat` using the authenticated redirect contract. Confirm the
   route builds as dynamic per request and leaves `/thread-chat` unchanged. Run
   `pnpm typecheck`.
4. Add the exact AGPLv3 text, package SPDX field, bilingual READMEs, and
   contribution guide. Verify all commands and links against the repository.
5. Implement landing CSS and components in reading order: header/hero,
   interaction steps, workspace, canvas, capabilities, closing CTA/footer.
6. Replace `app/page.tsx` composition and metadata; remove obsolete placeholder
   component contracts and unused imports.
7. Sweep landing files for duplicated copy, URLs, route literals, icon maps, and
   CSS values. Keep stable values in the approved constants/modules.
8. Run formatting only after implementation is complete, then execute the full
   validation matrix.

## 4. Validation Matrix

### 4.1 Automated repository checks

- `pnpm typecheck`
- `pnpm lint`
- `pnpm build`
- `pnpm openspec:validate`

### 4.2 Documentation and licensing checks

- Confirm `README.md` begins in English and links to `README.zh-CN.md`.
- Confirm both READMEs contain the live and GitHub URLs and equivalent feature,
  setup, architecture, roadmap, contribution, and license sections.
- Confirm GitHub/SPDX recognizes the unmodified license as AGPLv3.
- Confirm `package.json` remains private and declares `AGPL-3.0-only`.
- Confirm no README command uses npm or yarn for repository operations.

### 4.3 Route behavior checks

- Inspect rendered homepage links: every “Start chatting” points to
  `/start-chat`; every GitHub action points to the canonical repository.
- Signed-out `/start-chat` redirects to sign-in with `/start-chat` encoded as
  the return target.
- Two authenticated `/start-chat` requests produce different valid UUID tree
  URLs.
- Direct `/thread-chat` with remembered local state still resumes the last tree.

### 4.4 Visual and accessibility checks

- Desktop widths: 1440, 1280, and 1024 pixels.
- Mobile widths: 390 and 360 pixels.
- No horizontal document overflow.
- Correct heading order, link names, focus-visible states, contrast, and
  reduced-motion behavior.
- Landing styles do not alter `/thread-chat`, sign-in, account, or legal pages.

### 4.5 Regression checks

Run relevant existing Thread Chat E2E scripts when their database/model/browser
prerequisites are available. At minimum, run persistence and live-workbench
smoke coverage because the entry route feeds the same tree loader.

## 5. Completion Criteria

The change is complete only when the public repository has an English-first,
factually current README with a Chinese mirror; AGPL-3.0-only is consistently
applied; the landing page communicates the approved product story in the
research-notebook visual language; homepage starts always create fresh tree
URLs; the legacy resume route is unchanged; and the validation matrix passes or
any environment-only limitation is explicitly recorded.
