# 代码工作台·沙箱运行时调研报告

> 调研与讨论时间：2026-07-07 ~ 2026-07-09。
> 本文整理「代码工作台」（AI 实时生成 React/Next.js Demo）功能从运行时选型到规模化经济学的调研与决策过程。**实现细节、文件结构与迭代路线图见 [openspec/changes/add-code-workbench/](../../openspec/changes/add-code-workbench/)（design.md + tasks.md），随代码持续维护**；本文侧重记录推理过程本身——为什么选这个方案、算过的账、踩过的坑、与外部生态的关系判断——这些内容不会直接体现在代码里，但决定了后续迭代方向。

## 0. 摘要（TL;DR）

1. **产品目标**是做出 bolt.new / lovable 式的「对话即写码」体验：用户描述一个 React 组件/页面，右侧实时出预览 + 代码。
2. **运行时选型是两级火箭**：v1 用 **Sandpack**（浏览器内打包器，零服务端成本）覆盖纯前端 Demo；v2 用 **Apple container**（每容器一个轻量 VM，本机实验）跑真正的 `next dev`，验证「真 Next.js 全栈」路线，已于 2026-07-07 端到端跑通（VM 启动+文件同步+就绪约 12 秒，HMR 约 3 秒）。
3. **规模化不是"用户数 × 常驻沙箱"的算术题**：沙箱可以且应该被设计成可丢弃的计算缓存（源文件持久化在数据库，沙箱随时销毁重建），真实峰值并发通常只有日活的 5%~20%，客户端优先（Sandpack）路径能再把云端用量砍到 1/10。
4. **我们自研的 `SandboxRuntime` 协议，本质上是在做 [Sandbank.dev](https://sandbank.dev/)（一个面向 AI Agent 的开源统一沙箱 SDK）同一层的事**，只是覆盖面更宽（横跨浏览器和 VM，Sandbank 只做服务端）、深度更浅（没有快照/休眠/多云热切换）。这为将来「换成云端实现」提供了明确的抽象参照。
5. **托管云沙箱的市场价格惊人地收敛**：2 vCPU + 2GB 规格下，Daytona 与 E2B 报价完全一致（≈$0.133/沙箱·小时），这基本就是"managed Firecracker/KVM + 秒级启动 + API"这个品类的定价锚点；自托管（Fly.io 裸机、BoxLite 开源自托管）能便宜 1~2 个数量级，代价是自己扛运维。

## 1. 背景与目标

现有 thread-chat 是一个基于 assistant-ui + AI SDK v7 的聊天应用（MiniMax 模型）。目标是新增一种能力：用户说"帮我写一个基于 Tailwind CSS、framer-motion 的 Dialog 组件"，模型生成代码后，右侧打开一个在线预览器展示可交互的 Demo，并可切换到代码视图查看源码。最终对标 bolt.new、lovable 的产品形态，计划分多轮迭代逼近。

MiniMax-M3 是本项目的执行模型，实测单次工具调用能可靠输出 9KB+ 转义完好的多文件 TSX 代码，这是"代码走流式 tool args、服务端 execute 只回轻量确认"这一协议设计成立的前提。

## 2. 运行时方案调研与选型

预览/执行 Demo 代码需要一个"运行时"。调研时对比了五条路线：

| 方案 | 真 Next.js | 启动速度 | 接入成本 | 主要风险 |
|---|---|---|---|---|
| **Sandpack（react-ts）** | ❌ 纯前端 React | 秒级 | 极低（现成 React 组件） | 依赖 codesandbox.io 远程打包器；网络环境下其 Cloudflare 挑战会有握手毛刺 |
| Sandpack + Nodebox（nextjs 模板） | ⚠️ 旧版 Next、慢 | 10s+ | 低 | Nodebox 维护停滞，Tailwind v4 构建链不可用 |
| **WebContainers（StackBlitz）** | ✅ 浏览器内真 Node | 首次 10~30s | 中（COOP/COEP、进程编排、终端流） | 商用需授权；内存重；网络环境不确定性 |
| **本机 VM 隔离（Apple container）** | ✅ 完整 fidelity | pnpm 缓存下秒级 | 中（VM 生命周期管理） | 跑在开发者本机；不可直接部署到 serverless；见第 4 节踩坑 |
| 云沙箱（Vercel Sandbox / E2B 等） | ✅ | 秒级 | 中高（计费、API） | 付费；本阶段无部署需求 |

**v1 决策：Sandpack。** 理由：当天可跑通端到端链路，覆盖用户举例的全部场景（组件/动效 Demo 均为纯客户端代码）；管线各层（工具协议、状态管理、面板 UI、系统提示词）与具体运行时解耦，后续换运行时时全部可复用。

**v2 决策：本机 VM 隔离，优先于 WebContainers。** 理由：项目跑在开发者本机，用真实 VM（而非裸进程）跑 `next dev` 能给出 100% fidelity（App Router、Server Actions、API 路由、真 Tailwind v4 构建），同时比"AI 生成代码直接跑在宿主机进程"多一层隔离；WebContainers 的商用授权、内存开销与网络不确定性留作将来部署公网版本时的备选。选定的具体实现是 **Apple 开源的 `container`**（Homebrew formula `container`，Apple silicon 原生，每容器一个轻量 VM），因为设备恰好是 Apple silicon Mac，且系统版本（macOS 26 Tahoe）是该工具网络支持最完整的版本。

## 3. v1 实现要点（Sandpack）

- 代码走**流式 tool args** 而非 tool result：`createDemo` 工具的 `execute` 只回一个轻量确认 `{ok, fileCount}`，避免大段代码在模型上下文里往返两遍。
- **防御式文件规整**：模型输出不可控，需要让预览"尽量能跑"而不是报错——路径归一化、`@/` 别名 import 改写为相对路径（Sandpack 打包器不认 tsconfig paths）、缺 `/App.tsx` 时用首个组件文件兜底、内置 `cn()` 工具文件。
- **流式/完成两阶段**：生成期间 `autorun:false` 只喂代码编辑器（避免半成品代码反复触发打包报错），完成后通过 React `key` 重挂 `SandpackProvider` 触发一次干净构建，自动切到预览。
- 两个实测踩坑：① Tailwind v4 的工具类在 `@layer` 里，会被 Sandpack（stitches）的无 layer 样式压过，高度覆盖必须加 `!important`；② codesandbox.io 打包器域名的 Cloudflare 机器人挑战会让父层"加载完成"握手偶发丢失（iframe 内部其实已渲染成功），需要 CSS 隐藏冗余的加载遮罩。

## 4. v2 实验：Apple container 本机 VM 沙箱

### 4.1 架构

每个 Demo artifact 对应一个独立的轻量 VM（真实 `next dev` 进程），文件通过 `tar` 管道写入 VM 内文件系统（不做宿主 bind mount，VM 内原生 inotify 让 HMR 正常工作）。宿主浏览器通过 VM 分配到的私有 IP 直连预览端口。

### 4.2 实测数据

| 指标 | 数值 |
|---|---|
| 基础镜像构建（首次，含 npm install，国内走 npmmirror） | 约 2~5 分钟 |
| VM 启动 + 文件同步 + `next dev` 就绪 | 约 12 秒 |
| 修改文件后 HMR 热更新生效 | 约 3 秒 |
| 每个沙箱 VM 内存占用上限 | 2GB（`--memory 2g` 配置值，非必然占满；`next dev` 本身约 400MB~1GB） |

### 4.3 五个关键踩坑

1. **`container cp` 不支持 docker 的 `dir/.` 内容语义**（会把整个目录当子目录塞进去）。改用 `tar -cf - . | container exec -i <name> tar -xf - -C <dest>` 管道，目录树原样保留。
2. **Next 16 dev 默认拦截跨源 dev 资源**：宿主浏览器经容器 IP 访问对 `next dev` 而言是跨源请求，客户端 chunk / HMR 全部被拒——页面只有 SSR 骨架、无法水合（能看不能点）。修复：模板 `next.config` 启动时枚举 VM 自身对外 IPv4，动态填入 `allowedDevOrigins`。
3. **`container ls --format json` 的实测字段结构**：容器名在 `.id`（不在顶层 `status` 字符串），运行状态在 `.status.state`，IP 在 `.status.networks[0].ipv4Address`（CIDR 形式，需要 split `/`）——这套结构在 CLI 1.0 版本里未形成稳定文档化契约，需要宽容解析。且 **stop/start 后 IP 会变化**，重连时要重新发现。
4. **与 Docker Desktop 的 vmnet 网络互扰**：批量创建/销毁沙箱 VM 时，Apple container 动态管理 vmnet 接口的时机若恰好撞上 Docker backend 的网络枚举请求（`GET /networks`），曾导致 Docker daemon 整个死锁挂起（本项目的 Postgres 跑在 Docker 里，连带 API 500）。当时磁盘紧张（见下一条）也可能是诱因之一。恢复顺序：`container system stop` → 重启 Docker Desktop → 手动拉起被依赖的容器 → `container system start`。两者稳态共存正常，但要避免在 Docker 启动/枚举网络的同时批量操作沙箱 VM。
5. **磁盘空间敏感**：镜像 + builder VM（`buildkit`，独立占用约 2GB）+ 各沙箱可写层合计数 GB，磁盘可用空间 <2GB 时 `container run` 直接失败（`NSPOSIXErrorDomain Code=28`）。`buildkit` builder 不用时可 `container stop buildkit && container delete buildkit` 回收其占用。

### 4.4 遗留事项

- 修复 3 尚未重新烘焙进基础镜像（当前沙箱是运行时热修的，模板文件已更新，需要一次干净的 `container build` 落地）。
- 沙箱空闲自动回收未实现（当前需手动销毁）。

## 5. 规模化的资源经济学

**核心问题**：每个用户开一次新会话就要拉起一个沙箱，这在 100 / 1 万用户规模下是否是资源黑洞？

### 5.1 先纠正对"2GB"的理解

`--memory 2g` 是配置的**上限**，不是必然消耗。真实构成里 `next dev` 本身占大头（400MB~1GB），VM 层再加一两百 MB 开销。更重要的是架构特性：Apple container 是**一个容器 = 一个独立 VM**，VM 之间不共享内核（不像 Linux 上的 Docker 所有容器共享一个内核），密度天花板由此而来——16GB 的 Mac 上现实并发大概 3~5 个沙箱。这符合预期，因为它本来就是苹果给**开发者本机**做的工具，不是为多租户服务器设计的。

### 5.2 关键换算：用户数 ≠ 沙箱数

有两个杠杆能把"并发沙箱数"和"用户数"解耦：

- **杠杆一：沙箱只在"活跃编辑"时存在。** 源文件全部持久化在数据库（本项目里是 Postgres 的 `messages.content` JSONB），沙箱本身可丢弃、可重建（本机实验中 12 秒冷启动）。策略应是：空闲若干分钟即销毁，用户回来再重建。行业经验峰值并发 ≈ 日活的 5%~20%。
- **杠杆二：客户端运行时（Sandpack）零服务端成本。** 纯前端 Demo（占多数场景）跑在用户自己的浏览器里，这正是 bolt.new 整个商业模式的基础（WebContainers 同理全跑客户端）。只有真用到 API 路由 / SSR / 数据库的场景才需要升级到服务端沙箱。

### 5.3 规模阶梯

| 规模 | 方案 | 量级感受 |
|---|---|---|
| 1 人（当前，本机实验） | Apple container 本机 VM | 密度上限 3~5 并发，够用；可调低 `--memory` 到 1g |
| 10~100 用户 | 迁到 Linux 服务器：Docker/containerd + gVisor，或直接用托管沙箱（E2B、Vercel Sandbox、Fly Machines，按活跃秒计费、空闲归零） | Linux 容器共享内核，密度远高于 VM-per-container；托管方案完全免运维 |
| 1 万用户 | ① 客户端优先做默认预览，服务端沙箱只留给真全栈场景；② 服务端用 Firecracker microVM 舰队 + 快照恢复（E2B/CodeSandbox 底层同款，快照唤醒 <1 秒）+ 自动扩缩容 | 客户端优先能再砍 5~10 倍云端用量 |

### 5.4 驯化服务端沙箱成本的四个套路

1. **生命周期绑"活跃窗口"而非会话**——空闲即回收。
2. **Scale to zero**——停止即停费。
3. **快照休眠**——把整个 VM 内存状态快照落盘（磁盘价约为内存的百分之一），恢复可到亚秒级。这是 E2B / CodeSandbox / Cloudflare Sandbox 的 `sleep` / Sandbank 的 `snapshot` 能力的共同原理。
4. **可丢弃性**——沙箱本质只是**计算缓存**：真相和状态存在外部持久层，沙箱随时可以杀、随时可以重建。杀掉不是损失，是设计。

本项目的架构已经天然具备第 4 条（Demo 文件在数据库里，沙箱可重建），这是后续加空闲回收、甚至换成有快照能力的云供应商时最省事的基础。

## 6. 与 Sandbank.dev 的关系

### 6.1 Sandbank 是什么

[sandbank.dev](https://sandbank.dev/)（v0.2.0，MIT 开源）是一套面向 AI Agent 的**统一沙箱 TypeScript SDK**：一个接口，五个云适配器（`@sandbank.dev/{cloud,daytona,flyio,cloudflare,boxlite}`）。核心能力：

- 统一的 `create / exec / destroy` 生命周期，`port.expose` 暴露预览 URL，部分适配器支持 `snapshot`（快照）、`sleep`（休眠）。
- **能力探测**（capability detection）：`withStreaming` / `withPortExpose` / `withSnapshot` 等安全类型收窄，运行时判断当前 provider 是否支持某能力。
- 服务层扩展：`Db9ServiceAdapter` 给沙箱自动注入一个 serverless Postgres（凭据自动挂载为环境变量）。
- 多 Agent 编排：`createSession` + `spawn` + 共享 `context`，支持多个沙箱之间实时通信协作。
- 其 GitHub README 显示项目定位已从"统一沙箱接口"演进为"跨后端同步 Workspace 的 Agent Harness"，默认推荐后端是托管版 BoxLite（Sandbank Cloud）。

**需要澄清的命名混淆**：Sandbank（服务端沙箱 SDK）与 Sandpack（CodeSandbox 出品的浏览器内打包器/预览组件，本项目 v1 在用）是完全不同的两个项目，名字相似但没有关系，分别处在不同的层。

### 6.2 三层对比

| | Sandpack | Sandbank | 本项目的 SandboxRuntime |
|---|---|---|---|
| 本质 | 一个具体的**运行时** | 运行时之上的**统一协议 + 适配器**（只做服务端） | 运行时之上的**统一协议 + 适配器**（自研迷你版） |
| 跑在哪 | 用户浏览器，零服务端成本 | 云端（KVM/Firecracker/isolate） | 一个适配器在浏览器（Sandpack），一个在本机 VM（Apple container） |
| 核心接口 | React 组件 + files props | `create / exec / destroy / port.expose / snapshot / sleep` | `ensure / applyFiles / status / destroy / build` |
| 角色 | 是"被适配的对象" | 是"适配层本身" | 是"适配层本身" |

**结论：我们的 `SandboxRuntime` 干的不是 Sandpack 的事，而是 Sandbank 的事**——只是覆盖范围不同。接口逐项对照同构性很明显：`ensure` ≈ `create`，`applyFiles`（tar 管道写文件）≈ 用 `exec` 写文件，`status` 里的 IP 发现 ≈ `port.expose` 返回预览 URL，`destroy` 一一对应；连能力探测的思路都一致——Apple container 的 cp 语义差异、资源 flag 可能不存在时的降级重试，本质就是手写的能力适配。

### 6.3 各自多出来的部分

**Sandbank 有而我们没有**：`snapshot`/`sleep`（秒级唤醒，替代我们 12 秒冷启动）、`exec.stream`（流式日志）、多供应商热切换、DB9 服务层（给沙箱自动挂数据库，对"Demo 带数据库"场景有想象力）。

**我们有而 Sandbank 没有**：① 浏览器端这半边完全是 Sandbank 不管的范畴，而这是本项目多数场景的零成本默认路径；② Apple container 本地适配器；③ 产品层——文件规整、别名改写、`use client` 注入、面板状态机、artifact 持久化，这些永远是自己的事。

### 6.4 对本项目的实际意义

未来上云时，`SandboxRuntime` 的云端实现可以直接参考 Sandbank 的接口形状来定义（`create/exec/destroy` + 能力探测），届时甚至可以直接引入 Sandbank 本身，一次性接入它背后的五家云。**暂不直接引入依赖**：v0.2.0 太年轻，先观察其维护节奏，等真到"要上云"那一步再评估。

## 7. 接口统一性设计原则

浏览器沙箱（Sandpack：挂载组件即"创建"、更新 props 即"写文件"、卸载即"销毁"，无生命周期可言）与 VM 沙箱（有真实的启动/轮询/销毁过程）在"过程"层面差异巨大，但接口仍然可以统一，靠两个原则（Sandbank 自己的五个云适配器之间也是靠这两条统一的）：

1. **把接口定义成"幂等的意图"而非"过程"。** 例如 `ensure()` 的语义是"让预览存在"——Sandpack 实现同步返回、零成本；容器实现要跑十几秒。方法相同，成本不同，调用方不关心。
2. **用能力探测吸收差异**，而不是让每个适配器都实现全集接口：

```ts
interface SandboxRuntime {
  ensure(artifact): Promise<Preview>     // Sandpack: 即时返回；container: 慢
  applyFiles(files): Promise<void>       // Sandpack: setState；container: tar 管道
  destroy(): Promise<void>               // Sandpack: 卸载；container: 销毁 VM
  capabilities: Set<"serverSide" | "logs" | "snapshot" | "shell">
}
```

面板据 `capabilities` 决定是否显示"日志 / 销毁 VM"等按钮；浏览器适配器汇报空能力集，自然降级。

**一个克制的提醒**：目前只有两个实现，面板里简单的 if/else 已经够用，**不必急于把这个接口固化成正式抽象**——等第三个适配器（云端）真要接入时再抽，那时对语义的理解已经被三个实现锤炼过，抽象才不会画歪（对应 `openspec/changes/add-code-workbench/tasks.md` 里的 2.b.1，明确标记为"后续"而非当前任务）。

## 8. 云沙箱成本对比

### 8.1 前提假设

- 规格对齐目标负载：**2 vCPU + 2GB**（跑 `next dev` 的合理规格）。
- 使用模型：每个日活用户每天约 **30 分钟**活跃沙箱时间（空闲即回收）。100 DAU ≈ 1,500 沙箱·小时/月；10,000 DAU ≈ 150,000 沙箱·小时/月。
- 价格为 2026-07-09 官网抓取的实时数据，不含流量出口费；后续请以官网为准复核。

### 8.2 单价与月成本对比

| 供应商 | 折算 $/沙箱·小时（2C2G） | 100 DAU/月 | 10k DAU/月 | 备注 |
|---|---|---|---|---|
| **Sandbank Cloud** | $0.12（规格锁死 2C/**1G**，非 2G） | ~$180 | ~$18,000 | $0.02/10 分钟粒度计费；exec/文件操作免费；用 USDC 经 x402 协议支付，无需注册/API key；v0.2.0 极年轻 |
| **Daytona** | $0.133 | ~$200 | ~$20,000 | 按秒计费；新用户送 $200 额度（≈100 DAU 首月免费）；创业计划最高 $50k credit |
| **E2B**（参照） | $0.133 + 订阅 | ~$350（含 Pro 套餐 $150/mo） | $20k + 需谈企业版 | 费率与 Daytona 完全一致；Pro 套餐并发上限 100，10k DAU 高峰（约 1000 并发）不够用 |
| **Cloudflare Containers** | $0.05~0.11（对 CPU 活跃占比敏感） | ~$75~170 | ~$7k~17k | **内存/磁盘按预配（wall-clock）计费，CPU 只按活跃秒计费**，加自动 sleep 缩零；间歇型访问最省；需要基础 $5/mo Workers Paid |
| **Fly.io** | **$0.016**（shared-cpu-2x + 2GB） | ~$24 | ~$2,400 | 便宜一个数量级的代价：裸 IaaS，池化/回收/安全编排要自己写；停机的 Machine 只收 rootfs 存储费 $0.15/GB·月 |
| **BoxLite 自托管** | ~$0.002~0.003（满载摊销，非线性计费） | **~$70**（一台 64GB 独立服务器的固定租金，此规模下远未跑满） | ~$2,000~3,000（约 25~30 台服务器）+ 运维人力 | 软件本身开源免费；成本模型从"按量计费"变成"包机"，是台阶函数而非线性函数 |

### 8.3 三个关键读数

1. **托管沙箱服务的市场价格惊人一致：约 $0.133/沙箱·小时（2C2G）。** Daytona 和 E2B 费率完全相同（分别用官网的按秒单价折算验证过），Sandbank Cloud 的 $0.12（虽规格是 1G 而非 2G）也贴近这条线——这基本就是"managed Firecracker/KVM + 秒级启动 + API"品类的定价锚点。差异化在计费粒度（Sandbank 按 10 分钟切片）、并发上限（E2B 要额外购买）、附赠能力（Cloudflare 的自动休眠）。
2. **价格跨越两个数量级，本质是"自己扛多少运维"的滑杆。** 全托管（$0.133）→ Cloudflare 半托管（$0.05~0.11）→ Fly 裸机器（$0.016）→ 自托管 BoxLite（$0.003）。10k DAU 档：$20,000/月 vs $2,400/月 vs $2,000~3,000/月 + 一个人的运维心智，选择本质是"招不招人"的问题。
3. **客户端优先（Sandpack）是最大的折扣券。** 若 90% 的 Demo 走浏览器端渲染，云端用量直接砍到 1/10：100 DAU 的云端用量降到 ~150 沙箱·小时/月（Daytona/Sandbank Cloud 折算 ~$20/月，Daytona 的免费额度能覆盖近一年）；10k DAU 降到 ~15,000 沙箱·小时/月（托管方案 ~$2,000/月，Fly/自托管 ~$250~500/月）。

### 8.4 分阶段建议

- **验证期（≤100 用户）**：选托管服务——Daytona（白嫖 $200 额度 + 接口最正统）或 Cloudflare（若访问模式以间歇为主）。Sandbank Cloud 的 10 分钟计费粒度很适合短会话，但 v0.2.0 + 加密货币支付，列入观察名单。
- **上量后（>1k DAU）**：Cloudflare 的"CPU 只算活跃 + 自动休眠"与本项目"沙箱=可丢弃缓存"的架构最契合；预算敏感则选 Fly 或自托管 BoxLite，把省下的钱换成自己写编排层——而这层编排协议（`ensure/apply/status/destroy`），已经在 Apple container 实验里写过一遍，换后端本质是换实现，不是重新设计。
- **无论选谁**，"Sandpack 默认 + 服务端沙箱按需升级"这个开关都应该先落地，它比任何供应商选择都更影响成本结构。
