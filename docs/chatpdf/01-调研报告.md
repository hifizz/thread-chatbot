# ChatPDF 功能调研报告

> 调研日期：2026-07-05。GitHub star 数均为当日通过 GitHub API 实测。
> 本报告是 ThreadChat「附件上传 + ChatPDF」功能的前置调研，结论直接驱动 [02-设计方案.md](./02-设计方案.md)。

## 0. 摘要（TL;DR）

1. **单文档 ChatPDF 的主流路线已经分化为三条**：RAG（解析→分块→向量检索）、长上下文直塞（context stuffing）、页面级视觉（把 PDF 页当图片给多模态模型）。2025-2026 年的社区共识是：**单文档且总量 < 200K token 时直接塞全文即可，不必建索引**（Anthropic 官方工程指南的明确阈值），RAG 的价值区间在多文档/大语料库。
2. **PDF 解析是整条链路质量的天花板**，也是各开源项目差异化的主战场。JS/Node 生态里 2026 年服务端解析的事实首选是 **unpdf**（unjs 出品、零依赖、按页提取、serverless 友好）；复杂版式/扫描件需要 Python 侧 VLM 方案（MinerU、Docling）或托管 OCR API，属于二期能力。
3. **本项目的硬约束**（读 `ai@7.0.14` 与 `@ai-sdk/openai-compatible@3.0.5` 源码确认）：MiniMax 的 OpenAI 兼容端点只接受 `text` / `image_url` / `video_url`，**不接受任何 file content part**；且 openai-compatible provider 遇到「PDF file part + URL」会直接抛 `UnsupportedFunctionalityError`。**因此 PDF 必须在我们自己的服务端提取成文本后注入对话**，没有"把 PDF 直接给模型"的捷径。
4. **产品体验的胜负手不在模型而在 UX**：引用溯源（点击引用跳回原文页码）、上传/解析状态机、自动摘要 + 建议问题，是 ChatPDF.com / NotebookLM / kotaemon 等标杆产品反复验证过的三板斧。
5. 结论：ThreadChat 一期采用「**R2 私有桶直传 + 服务端 unpdf 按页提取 + 全文注入（带页码标记与截断保护）**」路线，把附件模块做成与文件类型解耦的通用基础设施；RAG（pgvector）、引用高亮、OCR 均作为明确的二期演进项保留扩展点。

## 1. 调研方法

- **开源项目盘点**：GitHub API 核验 star / 活跃度 / 语言构成，重点项目直接读源码（如 AnythingLLM 的 PDF 解析器、Vercel chatbot 的上传路由）。
- **技术路线**：以论文（NoLiMa、RULER、LOFT、Self-Route、ColPali）与厂商工程博客（Anthropic Contextual Retrieval、Chroma Context Rot）为一手证据。
- **本项目约束**：对照 `pnpm-lock.yaml` 锁定版本，逐行核对 `@assistant-ui/*`、`ai`、`@ai-sdk/openai-compatible` 的 dist 类型与实现源码（比文档可靠，assistant-ui 是快速迭代的 pre-1.0 包）。

**盘点结论**：

1. 通用型头部项目全部收敛到同一形态：**可换解析器 + 可换向量库 + 混合检索（BM25+向量）+ rerank + 引用溯源**。差异化集中在解析层与引用 UX。
2. 纯教学型单文档 chat-with-PDF 项目（mayooear、weaviate/Verba）已相继归档——**单文档 PDF 问答不再是独立赛道，而是聊天产品的一个标配功能**，这正是 ThreadChat 要做的形态。
3. JS 全栈做这件事完全可行（AnythingLLM 62.6k star 即证明），但 JS 规则解析只能覆盖「文本层完好、版式简单」的 PDF；复杂文档的高质量解析目前仍是 Python VLM 生态（MinerU 73.5k、Docling 62.7k）的主场。

## 3. 技术路线分析：RAG vs 长上下文 vs 视觉

### 3.1 长上下文直塞（context stuffing）

- **能力面**：Gemini 3 Pro 已到 1M token；Claude Sonnet 4.6 1M 且不加价；本项目所用 MiniMax M2 约 200K。单个 PDF 多数在几万 token 量级，完全塞得下。
- **Anthropic 官方工程阈值：知识库 < 200K token（约 500 页）直接整体放进 prompt，不必做 RAG**（[Contextual Retrieval](https://www.anthropic.com/engineering/contextual-retrieval)）。这是对"单文档 ChatPDF 是否需要 RAG"最权威的工程回答。
- **但要清醒认识长上下文的真实衰减**：Chroma [Context Rot](https://www.trychroma.com/research/context-rot) 实测 18 个前沿模型性能随输入长度单调下降；[NoLiMa](https://arxiv.org/abs/2502.05167) 显示去掉字面匹配线索后多数模型"有效长度"仅 ≤2K；Llama 4 Scout 标称 10M 窗口但 128K 长文任务准确率仅 15.6%。**标称窗口 ≠ 有效窗口**，塞全文时应把关键内容结构化（页码标记、目录）而非一锅粥。
- **成本**：全文塞入比 RAG 单查询贵 2-4 个数量级（量级参考：某对比给出 $0.10 vs $0.00008/查询）；但 **prompt caching 已改变经济性**——主流厂商缓存读取约为输入价 10%，同一文档多轮问答时"缓存整文"常优于建索引。

### 3.2 RAG

- "RAG 已死"论战（2025-10 [The RAG Obituary](https://www.nicolasbustamante.com/p/the-rag-obituary-killed-by-agents) 登顶 HN）的社区共识收敛为：**死的是 naive RAG，不是检索**。约 60% 生产级 LLM 应用仍用检索。
- RAG 的不可替代区间：**多文档/大语料库**。LlamaIndex 2026-01 基准：5 篇文档规模下文件系统 agent 检索准确度反超 RAG，但扩到 100-1000 篇后 RAG 在速度与准确率上均反超——**取决于语料规模**。
- 做好 RAG 的代价不小：Anthropic Contextual Retrieval 路线（chunk 加 LLM 生成的上下文前缀 + embedding + BM25 + rerank）能把检索失败率从 5.7% 压到 1.9%，但引入 embedding 供应商、向量库、rerank 三个新依赖。
- **本项目约束**：MiniMax 国际站目前无可用 embeddings 模型（中国站旧端点的 embo-01 非 OpenAI 格式），做 RAG 必须引入第二家供应商。Drizzle + pgvector 本身成熟（`drizzle-orm@0.45` 原生 `vector` 类型 + HNSW），路是通的，但不该在一期引入。

### 3.3 页面级视觉（vision-based）

- [ColPali](https://arxiv.org/abs/2407.01449)（ICLR 2025）跳过 OCR 直接把 PDF 页当图像做 late-interaction 检索，扫描件/图表场景效果显著；但多向量存储成本惊人（100 万页约 5.9TB vs 单向量 3.8GB），属于自建门槛很高的路线。
- 更现实的视觉路径是 API 原生：Claude 每页=图像+文本双通道、Gemini 每页固定 258 token。**MiniMax M2 无视觉能力，本路线一期不可用**，仅作为换模型后的演进方向记录。

### 3.4 路线决策

| 场景 | 推荐路线 |
|---|---|
| 单文档、< 200K token（本项目一期） | **全文注入 + 页码标记 + 截断保护** |
| 多文档 / 超长文档（二期） | pgvector 混合检索 + 第三方 embeddings |
| 扫描件 / 图表密集（二期+） | OCR 微服务或换视觉模型 |

## 4. PDF 解析：难点与选型

### 4.1 为什么难

PDF 是"打印排版格式"：内容是带坐标的 glyph 流，没有语义结构。典型失败模式：双栏论文逐行左右交错输出乱码；无标题结构的 200 页文档成"一团浆糊"；劣质 OCR 层把 `EUR 250` 读成 `EUR 25O`；表格（合并单元格/无边框/多级表头）公认最难。解析错误会向下游级联放大。

### 4.2 Node 服务端选型（一期范围）

| 库 | 状态（2026-07） | 判断 |
|---|---|---|
| **unpdf 1.6.2** | unjs 出品，零运行时依赖，内置 serverless 版 pdf.js（worker 内联、免 polyfill），`extractText(pdf, { mergePages: false })` 直接返回**按页字符串数组** | ✅ **一期选型**：按页提取天然支持页码标记，为二期"带页码引用"铺路 |
| pdf-parse v2 | 2025 年原作者回归重写，活跃；但引入 `@napi-rs/canvas` 原生二进制依赖 | ❌ 对 Next.js standalone/serverless 打包不友好 |
| pdfjs-dist 直用 | Mozilla 官方 | ❌ Node 下 worker/DOMMatrix 等 polyfill 坑多（unpdf 正是为解决这些而生） |
| mupdf.js | 提取质量最高 | ❌ AGPL（网络服务需开源或购买授权）+ WASM 14MB |
| Python VLM 系（MinerU 2.5、Docling） | OmniDocBench 90+ 分，已反超 GPT-4o/Gemini 等通用大模型 | 二期：复杂文档旁挂微服务或托管 API（Mistral OCR $1-2/千页、Unstructured $1/千页起） |

**清醒的预期管理**：unpdf（规则型文本层提取）只能覆盖「文本型、版式不复杂」的 PDF——这已经覆盖日常绝大多数场景；扫描件提取结果为空时应显式告知用户"该 PDF 无文本层，暂不支持"，而不是静默给出空上下文让模型幻觉。

## 5. 与模型/框架集成的硬约束（源码级结论）

这一节是本次调研最有价值的部分，全部结论对照本项目锁定版本的 node_modules 源码验证：

1. **附件 UI 已经全部就位**：`components/assistant-ui/thread.tsx` 已渲染 `ComposerAddAttachment`（加号按钮）、`ComposerAttachments`、拖拽 Dropzone 和 `UserMessageAttachments`。当前生效的是 react-ai-sdk 内置默认适配器 `vercelAttachmentAdapter`——把任何文件读成 **base64 data URL 内联进消息**。这意味着：**不接 R2 的话，附件会以 base64 整段写进 Postgres jsonb**，消息行体积爆炸，这正是要做对象存储的直接理由。
2. **接入点极简**：`AttachmentAdapter` 接口只有 `accept / add / send / remove` 四个成员，通过 `useChatRuntime({ adapters: { attachments } })` 传入即可覆盖默认适配器。`add()` 支持返回 AsyncGenerator 以 yield 上传进度（`status: { type: "running", reason: "uploading", progress }`）。
3. **附件内容的流转链路**：`CompleteAttachment.content` 里的 `FileMessagePart { data, mimeType, filename }` → react-ai-sdk `toCreateMessage` → AI SDK `FileUIPart { url, mediaType, filename }` → 持久化进 jsonb / 发给 chat route。回读时 `convertMessage` 把 `image/*` 的 file part 还原成附件缩略图。**`data` 字段放 URL 字符串即可，全链路透传**。
4. **PDF 不能直接给模型（决定性约束）**：
   - `@ai-sdk/openai-compatible` 对 `application/pdf` + URL 形态直接抛 `UnsupportedFunctionalityError("PDF file parts with URLs")`；
   - 即便转成 base64，产出的也是 OpenAI 专有的 `file` content part，**MiniMax 只接受 `text`/`image_url`/`video_url`，会 400**；
   - MiniMax 也没有可与 chat 互通的文件 API。
   - → **唯一正确做法：在 chat route 里、`convertToModelMessages` 之前，把 PDF file part 替换成服务端提取好的文本**。
5. **一个隐蔽的坑**：非 PDF 的其他 `application/*`（zip 等）同样会让 provider 抛错，`new URL()` 对相对路径 file part 也会炸——所以服务端必须对**所有** file part 做兜底转换（PDF→提取文本，图片→占位说明[M2 无视觉]，其他→附件元信息占位），保证任何附件组合都不会打断对话。

## 6. R2 存储选型要点

- **接入方式**：`@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` 走 S3 兼容 API；`region: "auto"`，endpoint `https://<account_id>.r2.cloudflarestorage.com`。
- **必加的坑规避**：aws-sdk v3.729.0+ 默认开启 flexible checksums，R2 兼容性事故频出（presigned 场景还会 `SignatureDoesNotMatch`）。**S3Client 必须配置 `requestChecksumCalculation: "WHEN_REQUIRED"` + `responseChecksumValidation: "WHEN_REQUIRED"`**——零成本的确定性做法。
- **直传 vs 中转**：presigned PUT 浏览器直传是主流（文件字节不过 Next.js 服务器，绕开平台请求体上限）；代价是要给桶配 CORS（`AllowedHeaders` 必须显式列出 `Content-Type`，R2 上 `"*"` 对 PUT 预检不生效）。
- **访问控制**：聊天附件是用户私有数据 → **私有桶 + 短时效 presigned GET**，不用 r2.dev 公开域。
- **安全基线**（OWASP File Upload Cheat Sheet）：MIME 白名单（而非黑名单）+ 分类型大小上限；`ContentType` 签进 presigned PUT（改了就 403）；解析前做魔数复验（PDF 查 `%PDF-` 前缀）；**对象 key 绝不含用户文件名**（`attachments/{uuid}.{白名单扩展名}`，原始文件名只进数据库），防路径穿越与枚举。

## 7. 成功核心点与衡量标准

### 7.1 产品层（商业产品验证过的三板斧）

| 模式 | 标杆 | 说明 |
|---|---|---|
| **引用溯源** | NotebookLM、kotaemon、Adobe AI Assistant | 编号引用 → 点击跳回原文页并高亮。信任的核心，也是 ChatPDF 类产品留存的胜负手。Anthropic Citations API 实测比提示词方案引用召回 +15% |
| **状态机可见** | RAGFlow | 上传→解析→就绪每一步显式可见；RAGFlow 甚至把"观看解析过程、逐 chunk 修正"做成卖点 |
| **冷启动引导** | ChatPDF.com、NotebookLM | 上传即自动摘要 + 3 个建议问题，消除"我该问什么"的空白页恐惧 |

### 7.2 技术难点排序

1. PDF 解析质量（天花板，见第 4 节）
2. 模型集成约束（本项目已查清，见第 5 节）
3. 长文档的上下文预算管理（截断策略、多附件配额分配）
4. 大文件的异步化（社区经验：50+ 页就应转异步任务队列；一期用「上传时同步解析 + 状态可见」覆盖到中等规模）

### 7.3 衡量标准

- **质量**：TruLens RAG Triad（上下文相关性 / groundedness / 回答相关性）；RAGAS 四指标是社区事实标准。一期最低要求：模型引用的内容必须真实存在于 PDF（人工抽查），无文本层时显式失败而非幻觉。
- **性能**：TTFT < 600ms 可接受（流式必做）；解析在上传阶段完成并入库，**对话阶段零解析开销**（避免每轮重复下载+解析）。
- **健壮性**：任意附件组合（含不支持的类型）都不得导致对话 500；上传/解析失败有明确的用户可见错误。

## 8. 结论：ThreadChat 的路线选择

**一期（本次实现）**：

1. 通用附件基础模块：R2 私有桶 + presigned PUT 直传 + MIME 白名单策略表（PDF/图片/ZIP/视频，每类独立大小上限与"模型可见性"策略）——新类型 = 策略表加一行。
2. ChatPDF：上传时服务端用 unpdf **按页**提取文本存入 Postgres（一次解析、多轮复用）；对话时在 `convertToModelMessages` 前把 PDF part 替换为带页码标记的全文（带全局字符预算与截断提示）。
3. UX：上传进度、解析状态、失败可见；无文本层 PDF 显式报错。

**二期扩展点（设计时预留，不实现）**：pgvector RAG（附件表已按页存储，chunk 化即可）、引用页码点击跳转（react-pdf-highlighter）、OCR 微服务、自动摘要+建议问题、prompt caching。
