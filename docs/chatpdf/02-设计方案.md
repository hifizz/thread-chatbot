# 附件上传 + ChatPDF 设计方案

> 前置阅读：[01-调研报告.md](./01-调研报告.md)。本文是一期实现的设计说明，随代码一起维护。

## 1. 目标与非目标

**目标（一期）**

1. 通用附件上传基础模块：Cloudflare R2 私有桶 + presigned PUT 浏览器直传；MIME 白名单策略表驱动，PDF 先行，图片/ZIP/视频挂号占位。
2. ChatPDF 端到端可用：上传 PDF → 服务端按页提取文本入库 → 对话时注入模型上下文 → 模型基于 PDF 内容回答。
3. 任意附件组合不打断对话（不支持的类型对模型降级为元信息占位，而不是 500）。

**非目标（二期扩展点，设计上已预留）**

- RAG / pgvector 检索（附件文本已按页存储，chunk 化即可接上）
- 引用页码点击跳转、PDF 预览高亮
- 扫描件 OCR、自动摘要 + 建议问题

## 2. 总体架构

```
┌─ 浏览器 ─────────────────────────────────────────────┐
│ Composer(assistant-ui, 已有 UI)                       │
│   └─ r2AttachmentAdapter (lib/chat/attachment-adapter)│
│        add():  POST /api/attachments  ── 校验+建行+presign
│                PUT <presigned URL>    ── 直传 R2（XHR 报进度）
│                POST /api/attachments/:id/ingest ── 触发提取
│        send(): content=[{type:"file", data:"/api/attachments/:id", …}]
└──────────────────────────────────────────────────────┘
          │ 消息 parts 里只存稳定的应用内 URL（不存 presigned URL）
          ▼
┌─ 服务端 ──────────────────────────────────────────────┐
│ /api/attachments            POST  校验策略→attachments 建行→presigned PUT │
│ /api/attachments/[id]       GET   302 → presigned GET（私有桶读取）        │
│                             DELETE R2 对象 + DB 行                        │
│ /api/attachments/[id]/ingest POST 魔数校验→unpdf 按页提取→存 pages/status │
│ /api/chat                   POST  resolveAttachmentParts():               │
│                                   pdf→<attachment> 包裹的按页全文          │
│                                   image/*→占位(M2 无视觉) 其他→元信息占位   │
│                                   → convertToModelMessages → MiniMax      │
└──────────────────────────────────────────────────────┘
          ▼
   R2 私有桶（原始文件）+ Postgres attachments 表（元数据 + 提取文本）
```

## 3. 关键设计决策（与调研结论的对应关系）

| 决策 | 理由（调研报告章节） |
|---|---|
| presigned PUT 直传，不走 Route Handler 中转 | 文件字节不过 Next 服务器，绕开平台请求体上限（§6） |
| S3Client 固定 `WHEN_REQUIRED` ×2 checksum 配置 | aws-sdk v3.729+ 与 R2 的 flexible checksums 兼容性事故（§6） |
| 私有桶 + 302 到短时效 presigned GET，消息里只存 `/api/attachments/:id` | 附件是用户私有数据；presigned URL 会过期，不能落进持久化消息（§6） |
| 上传时提取（ingest），不在对话时提取 | 一次解析、多轮复用；对话阶段零解析开销（§7.3） |
| unpdf + `mergePages: false` 按页存储 | 页码标记提升长文注入效果，且为二期 RAG/引用跳转铺路（§4.2） |
| chat route 对**所有** file part 兜底转换 | openai-compatible 对 PDF-URL 抛异常、MiniMax 不接受 file part、zip 等类型同样炸（§5.4-5.5） |
| 策略表 `ATTACHMENT_POLICIES` 驱动一切校验 | 新类型 = 加一行（kind + 上限 + 模型策略），客户端/服务端共用（§6 安全基线） |
| key = `attachments/{uuid}.{白名单扩展名}`，原始文件名只进 DB | OWASP：防路径穿越/枚举（§6） |
| 无文本层 PDF → ingest 标记 `failed` 并报错 | 显式失败优于静默空上下文导致幻觉（§4.2、§7.3） |

## 4. 数据模型

`lib/db/schema.ts` 新增：

```ts
attachments {
  id          text PK            // crypto.randomUUID()，同时是应用内 URL 的路径段
  key         text UNIQUE        // R2 对象 key：attachments/{uuid}.{ext}
  filename    text               // 原始文件名，仅展示用
  mimeType    text
  size        integer            // 字节；ingest 时与 R2 实际大小复验
  kind        text               // document | image | archive | video（来自策略表）
  status      text               // uploading | ready | failed
  pageCount   integer?           // PDF 专用
  pages       jsonb?             // string[]，pages[i] = 第 i+1 页文本（PDF 专用）
  error       text?              // 失败原因（用户可见）
  createdAt   timestamptz
}
```

按页存 `jsonb` 而非合并大文本：二期做 RAG 时直接对页/chunk 建 embedding，无需重新解析。

## 5. API 约定

| 路由 | 方法 | 请求 → 响应 |
|---|---|---|
| `/api/attachments` | POST | `{filename, contentType, size}` → 策略校验 → `{id, uploadUrl}`；越限返回 4xx + 中文错误信息 |
| `/api/attachments/[id]/ingest` | POST | 直传完成后调用。HeadObject 复验大小 → PDF 做 `%PDF-` 魔数校验 + unpdf 提取 → `{status, pageCount?}`；提取失败置 `failed` 并删 R2 对象 |
| `/api/attachments/[id]` | GET | 302 重定向到 1h 时效 presigned GET（图片缩略图 / 用户下载共用） |
| `/api/attachments/[id]` | DELETE | 删 R2 对象 + DB 行（composer 里移除附件时调用） |

## 6. 对话注入策略（`lib/chat/resolve-attachments.ts`）

在 `app/api/chat/route.ts` 里、`convertToModelMessages` 之前对 UIMessage 逐 part 处理：

- `application/pdf`：按 URL 提取附件 id → 读 DB `pages` → 组装
  `<attachment name="..." pages="N">\n[第 1 页]\n...\n</attachment>`
  文本 part。全局字符预算 `ATTACHMENT_CONTEXT_CHAR_BUDGET`（默认 12 万字符，MiniMax 200K token 窗口下的保守值），多附件平摊；超预算从尾部截断并追加 `[已截断：原文共 N 页，此处仅包含前 …]` 提示，让模型知情。
- `image/*`：替换为 `[用户上传了图片：xxx.png]` 占位文本（MiniMax-M2 无视觉；换视觉模型时改此分支即可）。
- 其他类型：`[用户上传了附件：xxx.zip（application/zip, 1.2MB）——该类型暂不支持内容解读]`。
- DB 查不到 / `failed` 状态：占位说明解析失败，不打断对话。

## 7. 客户端适配器（`lib/chat/attachment-adapter.ts`）

实现 assistant-ui `AttachmentAdapter`（覆盖默认的 base64 内联适配器）：

- `accept`：由策略表拼出（`application/pdf,image/png,...`）。
- `add()`：AsyncGenerator。本地校验（类型/大小）→ POST 建行拿 presigned URL → XHR PUT 直传（`progress` 实时 yield `{type:"running", reason:"uploading", progress}`）→ POST ingest（PDF 解析期间同样处于 running 态）→ yield `requires-action`。上传结果（id/状态）存模块级 Map，供 `send()` 消费；失败 throw，assistant-ui 会展示错误态。
- `send()`：等待 add 阶段的上传 promise 完成，返回 `CompleteAttachment`，`content: [{ type: "file", data: "/api/attachments/{id}", mimeType, filename }]`。
- `remove()`：DELETE `/api/attachments/{id}`（尽力而为，失败不阻塞 UI）。

选择「选中即上传+解析」而非「发送时才上传」：用户输入问题的时间与上传解析并行，发送零等待——这是 ChatGPT/Claude 的交互模式（调研报告 §7.1）。

## 8. 配置与环境变量

```
R2_ACCOUNT_ID=            # Cloudflare 账户 ID
R2_ACCESS_KEY_ID=         # R2 API Token 的 S3 凭证
R2_SECRET_ACCESS_KEY=
R2_BUCKET=thread-chat-attachments
```

- 桶 CORS 需允许来源域的 `PUT`，`AllowedHeaders` 显式含 `Content-Type`（见 README 示例）。
- 环境变量缺失时：`/api/attachments` 返回 503 + 明确的中文提示（"未配置 R2 存储"），聊天主流程不受影响。

## 9. 风险与已知限制

1. **扫描件/无文本层 PDF**：提取为空 → 显式失败。二期接 OCR。
2. **超大 PDF**：一期上限 20MB（策略表），ingest 同步解析在此规模下秒级完成；更大文件需异步队列（调研 §7.2）。
3. **presigned GET 时效**：图片缩略图 URL 经应用路由 302，每次请求重新签发，无过期问题；但外链复制出去 1h 后失效——符合私有数据预期。
4. **多会话并发覆盖**：附件行无用户体系（本项目无鉴权），与现状的 threads 一致，不在本期引入。
